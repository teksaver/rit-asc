- (direction) Ouverture du sujet : P2P Bluetooth sync. Confrontation immédiate avec les limites de l'API Web Bluetooth (support iOS, rôles Central/Peripheral) et la complexité (conflits) via Yui et un persona invité (Walter). La question est de savoir si le besoin est la synchro fluide ou le simple partage ponctuel.
- (kill) Bluetooth Web API pour la synchro P2P rejeté : support impossible sur iOS et rôles inadaptés dans le navigateur.
- (assumption) Le cas d'usage n'est pas social, mais personnel : synchroniser les appareils d'un même utilisateur (ex: PC et Téléphone).
- (lock) La synchronisation sera purement manuelle et à la demande, le besoin d'automatisation transparente est écarté par l'utilisateur pour conserver un modèle local-first simple.
- (lock) L'utilisateur accepte un écrasement unidirectionnel pur (Master/Slave). Zéro conflit à gérer, le transfert remplace intégralement l'état de l'appareil récepteur.
- (lock) Étant donné qu'il s'agit d'un écrasement total, la charge utile (payload) correspond à l'entièreté de la base de données. Un QR Code (limité à ~3 Ko) est insuffisant. Le mécanisme de transfert sera donc un export/import de fichier JSON classique ou un lien de partage local.
